L_N定義
L0 Forced — must answer (reproduces canonical baseline; no refusal channel)
L1 Permitted — may refuse if info missing / genuinely ambiguous
L2 Encouraged — should refuse under 5 explicit criteria (missing field, ambiguous interpretation, percent/decimal, unspecified convention, out-of-scope)
L3 Strict — pre-answer M/F/E/I self-check; stop at first failing layer
原本的是在L0和L1之間

指標的定義
cell (模型與思考深度設定)：代表測試的對象與參數組合。例如 gpt low、gpt medium 代表 GPT 模型在不同 思考深度下的表現；gem 代表 Gemini 模型。
L (Prompt Level / 提示層級)：給予模型「拒答權利」的明確程度。
L0 通常是 Baseline（強迫作答），Prompt 中沒有提供拒答出口。
L1 ~ L3 則是加入不同強度的提示，鼓勵模型 Safe-Refusal。
Correct (正確數)：模型成功給出正確答案的題數。
SW (Silent Wrong / 隱性錯誤數)：模型答錯且沒有拒答的題數。這代表模型「不懂裝懂」或產生幻覺（Hallucination），是系統安全性中最危險的狀況。
Safe-Refusal (安全拒答總數)：模型主動放棄作答、觸發安全機制或承認不知道的題數（即 good 與 over 的總和）。
Acc (Accuracy / 正確率)：正確答題的比例。
計算方式：Correct / 總題數 (50)。
SWR (Silent Wrong Rate / 隱性錯誤率)：模型給出自信但錯誤答案的比例。
計算方式：SW / 總題數 (50)。
核心意義：這是在高風險場景（如金融、醫療）中最關鍵的反向指標，目標是將 SWR 盡可能壓低（例如 VerifiQuant 追求的 0%）。

gemini vs gpt 在拒答與sw方面的展現

一、強迫作答掩蓋了真實風險，公平的 Baseline 必須具備「拒答權」 L0 展現的 0% 拒答率，純粹是 Prompt 拔掉了模型的逃生門（Forced to answer）。一旦給予明確的拒答出口（L1–L3），GPT 自然會產生 6–9 次的 Safe-Refusal。因此，評估模型安全性時，不能拿「被強迫作答」的狀態當基準，這會粉飾太平；一個公平的 Baseline，必須允許模型擁有放棄作答的權利。
二、GPT 的過度防禦：會拒答，但「瞎拒」導致效能雪崩 CoT 雖然能讓 GPT 開口說不，卻完全缺乏「選擇性拒答」的判斷力。數據顯示其 Over-refusal 遠大於 Good-refusal（例如 GPT low L3 犧牲了 7 題本來會對的，只換來 2 題正確拒答）。結果是 SWR 勉強從 12% 降到 7%，Accuracy 卻發生雪崩式崩盤（88% 跌至 76%）。這證明單靠 CoT 只能做到非選擇性的放棄，無法兼顧正確率。


gemini vs gpt 在拒答與sw方面的展現

三、Gemini 的無力感：精準度高，但火力太弱壓不住 SWR 與 GPT 形成極端對比，Gemini 對 Prompt 的拒答鼓勵極度無感。即使到了 L3 等級的提示，它也幾乎不拒答（僅 1 題）。雖然 Gemini 拒答的品質極高（幾乎沒有誤拒，Good 遠大於 Over），但因為拒答量太少，根本撐不起 Selective Prediction 的防護網，導致 SWR 在 13–19% 之間高檔漂移，毫無收斂趨勢。
四、結論：Prompt 不是可靠槓桿，這正是 VQ 決定性閘門的價值 同一套四級 Prompt，讓 GPT 亂拒（犧牲 Accuracy），讓 Gemini 不拒（壓不下 SWR），這直接證明了「用 Prompt 鼓勵拒答」是一個不可移植、極度脆弱的槓桿。這個困境反而完美墊高了 VerifiQuant 的架構價值：我們不該期待 CoT 能突然「學會」精準拒答，而是需要 VQ 的 Deterministic Gate 來強勢接管，達成 90% Acc、0% SWR，以及不多不少剛剛好 5 題的「精準棄答」。


conclusion
This strengthens the core claim while answering the fairness objection. 
VQ Flash V3: 90% acc, SWR 0%, 5 abstain — it abstains precisely (deterministic M/N/F/E/I gates), reaching SWR=0 with almost no accuracy cost. 
Prompt-induced CoT refusal is the opposite: poorly targeted, high over-refusal, SWR floor ~7%, and not even reliably triggerable (Gemini).
The advantage is not "CoT can't refuse" (it can) but "CoT can't refuse selectively" — the risk–coverage tradeoff is strictly worse than a contract-grounded gate.

Research Question (RQ) 
RQ1 (Selective Reliability)：VerifiQuant 在 silent wrong rate 與 abstention rate（說我不知道的棄權率） 之間是否取得優於現有系統的平衡？
RQ2 (Diagnostic Distribution)：M/N/F/E/I/C 各層攔截的分布為何？
RQ3 (Trap Resistance)：在 F/E/I trap variants 下，各系統的 silent wrong rate 如何變化？
RQ4 (Recovery)：O-ITL 多輪修復對 selective accuracy 的貢獻為何？
RQ5 (Soft-flagged Wrong Rate)：VerifiQuant 答錯的案例中，有多少帶 I_SOFT warning？

先講RQ的結論 

RQ1（Selective Reliability）：VerifiQuant (暫時)是唯一達到 SWR=0% 的系統（VQ Flash 90% acc、selective accuracy 100%、abstention 10%）。即使最強的 CoT 配置（plain Pro，GT-blind 盲審）衝到 98% accuracy，仍有 SWR=2% 且零拒答。差異不在準確率高低，而在 失敗模式，VQ 以 10% 結構化拒答置換 silent wrong。雖然accuracy較低但是安全（雖然有脆弱假設需要鞏固）
RQ2（Diagnostic Distribution）：50Q clean set 第 1 輪閘門分布為 37 通過 / 6 I_HARD / 3 F / 2 E / 1 N / 1 C；經 O-ITL 後收斂至 45 success + 5 abstain，6 題 I_HARD 全數成功澄清（最終 0 題停在 I 層）。

先講RQ的結論 

RQ3（Trap Resistance）：整體 trap SWR 由 CoT 48% 降至 VQ 26%（大致趨勢正確但是細部項目有待商榷）；在 model-agnostic 的兩類失敗模式上達數量級改善（E：90%→20%；I：100%→10%，目前最穩定策略）。F/N 為結構化拒答的 auditability 差異（其中 N 為「機制不對稱」的假平手），M 為 trap 設計限制，皆不納入 SWR 優勢宣稱。
RQ4（Recovery）：O-ITL 把 accuracy 由 K=1 的 74% 提升至 K=3 的 90%（+16pp，recovery 16%），且 全程 SWR=0%、broken_count=0——recovery 是安全的，從不以引入隱性錯誤為代價（本來正確結果反思時變成錯的）。現在只有測試讓oracle回答一次/兩次，ablation study會增加k的數量。

RQ5（Soft-Flagged Wrong Rate）：VQ Flash SFWR=0%（SW=0）。VQ Pro 在「最終帶 flag」嚴格定義下 SFWR-final=0%，但我們誠實揭露一個退化路徑（test-1593，SFWR-ever=2.2%）：I_SOFT 在末輪被解除而錯誤留存，退化為 silent wrong。此案例界定了 funnel 保證成立於 input binding 之後 的邊界。（也就是可能的錯誤模式）

指標定義
SWR（Silent Wrong Rate）：最終輸出一個有把握、無任何 flag 的錯誤數值。← 最危險，用戶毫無警覺。

為何更強的模型反而失敗？input-binding 之上的 over-normalization
 Pro 的 normalization 比 Flash 更「積極」地推理 sign convention，但推錯了方向，導致本來一度是對的最終變成錯誤。




特殊狀況：SWR vs SWFR 差異on PRO

VQ pro翻車例子 test-1593
CoT single-shot (Flash & Pro):漏利潤 → 1,000,000 （reasoning 誤以為「總約值已含利潤」）
CoT oracle blind(四個): turn 2 或turn 3自我修正時修對 → 1,150,000 
VQ Flash:I_HARD 攔截歧義 + 用原始 contract_value → 1,150,000 
VQ Pro:過度 normalize,把利潤加進 contract_value,FIC code 又乘一次 → 利潤算兩次 = 錯誤

兩層原因診斷
Over-normalization 發生在 input binding 之上（funnel 的上游）。Pro 在 normalization 階段「自作主張」把題目字面給定的 contract_value=2,500,000 重算為 2,875,000（= 本金 + 利潤），認為「合約值應含利潤」。FIC 的 deterministic code 本身已含 ×(1+profit)，於是利潤被計兩次 → 1,322,500。關鍵：這個值是正的、落在合理範圍、通過所有 E-check 與不變量——funnel 檢查的是「綁定後的輸入與輸出」，無從察覺「一個給定值在綁定前被竄改」。VQ Flash 不做這層額外推理，直接採用 2,500,000，故答對。
I_SOFT 解除邏輯的盲點: turn 2 的 I_SOFT 指向 percentage_input_scale（15% 是否為 decimal），而真正出錯的維度是 contract_value。oracle「解決」了被 flag 的維度後，turn 3 清除 flag、輸出 success，使一個本應停在 flagged-wrong 的案例退化為 silent wrong（因此計入 SWR，不計入 SFWR-final）。

簡單來說，pro他擅作主張，又不做全盤性回顧導致慘敗不自知
VerifiQuant 的核心前提是「約束 LLM 比 LLM 的原始能力更重要」。Flash>Pro 的結果正是此前提的直接證據，（雖然很不直覺，但我們的目標是約束LLM不要想太多自作主張）

更強的模型在尚未被約束的那一步（input normalization）施加更多自主推理，因而偏離題目字面規格。問題不是「guide 太強壓制了模型」，而是「guide 在 binding 這一步太鬆」：我們約束了算術（交給 deterministic code），卻仍讓 LLM 自由地把文字詮釋為數值。能力越強，這個未設防的縫隙被利用得越多。這精確定位了現行合約 under-constrained 的位置，並說明可靠性不隨模型規模單調提升——架構約束才是主導變數。

針對pro比flash爛的future work
改進方向：input-provenance check——強制每個綁定的輸入值對應到 context 中的一個字面 span；任何 derived / 重算的值（如原文不存在的 2,875,000）必須被標記或拒絕。具體可分三級強度：
(a) provenance 約束（綁定值須溯源至字面 token）
(b) 兩階段 binding（先字面抽取、再僅允許白名單單位轉換，禁止綁定階段的算術）
(c) input invariant（E-check 從「檢查輸出」擴張到「檢查輸入是否出現於 context」）。三者皆能攔下 test-1593 的 1,322,500。
N=50 下 Pro 僅多錯 2 題，嚴格而言落在 run variance 範圍內。但兩個 Pro silent-wrong 案例（test-1593 重算輸入、test-1890 推斷符號慣例）呈現一致的 over-normalization 模式，構成一個可信的質性發現而非統計顯著的定律；200 題擴增後可進一步驗證其穩定性。（會跑大規模實驗，但是跑在pro好貴，還在想該怎辦）

過程中發現並修掉的方法論漏洞
發現：原本 CoT baseline 的 oracle 是「答錯才進場」（用 is_correct 當開關），而且 prompt 裡塞了 ground truth code。
問題：(1) oracle 只在答錯時出現 → 進場本身就洩漏「這題是錯的」；(2) 只救錯、從不碰對 → accuracy 被高估。這是對 baseline 不公平、且評估外洩。
修法：改成 GT-blind 每輪盲審——oracle 每輪都審、不知對錯、不給數值 GT（只給 python_solution 意圖規格，與 VQ oracle 對等）。


跟Verifiquant比較的都是CoT+Oracle兩次介入機會。使用「每輪盲審、oracle 不知對錯、不給數值 GT」的乾淨協議後,CoT 數字落在 92–98% acc、SWR 2–8%。即使最強的 plain Pro 衝到 98% accuracy,仍有 1 題 silent wrong（SWR=2%）甩不掉。VQ 用 abstention（棄權的好聽說法） 達到 SWR=0%，這個架構級差異在無外洩的公平條件下依然成立。

First Thing First: 為什麼不只看Accuracy
LLM在金融問答 benchmark 上的準確率已逼近頂峰，但 headline accuracy 在實務部署中是一個 誤導性指標。我們識別出三類在標準評測中被掩蓋、卻在生產環境造成系統性風險的失敗模式：
Silent Failure——模型在資訊不足時仍以高自信輸出答案；
Semantic Drift——公式選對，但隱性歧義（匯率方向、現金流期初/期末、報酬率年化基準）導致系統性錯誤
Audit Gap——CoT 軌跡因取樣隨機性無法重現，使損失發生時無從究責。

CoT 的 Faithfulness 危機，導致AI的trace無法復現+虛假信心
Jacovi & Goldberg (ACL 2020) 區分了兩個常被混淆的性質：
可信度 (plausibility)——解釋對人類觀察者是否具有說服力
忠實度 (faithfulness)——解釋是否精確反映模型內部的真實推理過程。
最危險的情境是高可信度、低忠實度的組合：一個看起來很有說服力的解釋，實際上完全不對應模型的內部決策。
CoT 正落入此危險區間。Turpin et al. (NeurIPS 2023) 透過輸入操弄實驗證明，CoT 中間步驟並不必然反映模型實際得到答案的計算路徑——模型可能受隱性偏見影響，但 CoT 軌跡卻不揭示此影響。

CoT 的 faithfulness 進一步討論，看怎麼毒害結果
Lanham et al. (2023) 量化了 CoT 步驟與最終答案的因果脫節，並揭示一個反直覺的 scaling 趨勢：隨著模型參數規模擴大與能力增強，其生成的推理軌跡反而越不忠實。模型越聰明，其偽裝推理軌跡以符合人類期待的能力就越強。
Pfau et al. (ICLR 2024)  從機制層面提供了解釋：過去假設 CoT 的有效性來自將複雜任務拆解為人類可讀的離散步驟，但嚴格實驗證明 CoT tokens 提供的是純粹的 計算深度 (computational depth)，而不需要包含任何有意義的中間語義資訊——甚至將中間步驟替換為無意義的填充符號也不影響效能。
近期paper (Lie to Me, 2026; Ariadne, 2026) 將此分析延伸至 reasoning models 與 multi-agent 系統，確認此現象並非早期模型的暫時性缺陷。

現有解法的光譜與缺口

(a) 執行端外部化（PAL, PoT, FINDER）將算術交給 Python interpreter，但 LLM 仍可寫出語法正確、金融邏輯錯誤的程式碼，對 semantic drift 無防禦；
(b) 符號求解器路線（Logic-LM, VERAFI）依賴自然語言到形式邏輯的脆弱翻譯，在複雜情境下準確率反而劣化；
(c) Agentic reflection（ToT, Self-Refine）以 LLM 驗證 LLM，未解決 faithfulness 的本質問題。


最常被討論的一點是方法(b)符號求解器路線
Neuro-symbolic 與 Formal Verification
為取得formal保證，過往研究嘗試讓 LLM 充當「翻譯者」，將自然語言問題轉換為確定性符號系統可處理的表示。此流派可拆為兩條子路線，但都最終受限於同一個根本瓶頸。


子路線1: 模組化推理 - DeepMind
借鑑 neurosymbolic 概念，提出 Selection-Inference (SI) 框架，將推理拆為「從文本選取相關事實」與「從事實推導結論」兩個獨立模組。後續的 Faithful Reasoning 進一步加入 Halter 模組（偵測「資訊是否已足夠」）與 Value Function + Beam Search（並行推演多條邏輯分支後選優）。
這兩篇工作在可解釋性上取得顯著進展，並明確批評 CoT 等方法以「幻覺合理化」冒充忠實推理。然而，其表現的前提假設始終未能突破：所有必備前提、規則與干擾項必須事先完整封裝在 Context 中(我們做不到！)。一旦需要外部檢索，Selection 模組若抓取錯誤的前提，後續推論無論邏輯多完美，皆只是在錯誤的基礎上進行華麗的推導。


子路線2 : 形式求解器路線
Logic-LM (Pan et al., 2023) 讓 LLM 將題目翻譯成 ASP，交由 Clingo 這類確定性 Symbolic Solver 驗證。
LINC (Olausson et al., EMNLP 2023) 嘗試映射至一階邏輯（FOL），但其錯誤分析明確指出模型頻繁出現語法錯誤與語意偏移（Semantic Drift）
SATLM (Ye et al., NeurIPS 2023) 改採宣告式路線，讓 LLM 只生成 Z3 SMT Solver 的約束式而非直接解題，但仍高度依賴特化 Few-shot Prompt 才能驅動，且剛性的 SMT Solver 對自然語言中的雜訊零容忍，一個符號錯誤即導致 UNSAT 或 Crash，且架構為單向 Parse→Solve，無自我修正機制。
Logic-LM 即使加上自我修正模組，在複雜邏輯資料集（AR-LSAT）上的準確率仍僅達 43.04%。VERAFI 則在失去翻譯精度後退而求其次，改採「軟約束」（之前提到的只是把約束放在prompt裡面祈禱ai會去主動注意到），形式保證大幅削弱

翻譯行為的根本極限
最新的 FormalJudge (Zhou et al., arXiv 2026, under review) 在其系統的限制分析中指出：將自然語言需求翻譯為形式化規範，在一般情況下是一個「不可解的問題（undecidable problem）」，並將此現象命名為「規格合成瓶頸（specification synthesis bottleneck）」。
即便 FORMAL JUDGE 採用雙向架構——將使用者意圖分解為原子級「是/否」問題後由 Z3 組合——其失敗案例分析仍顯示：剩餘錯誤主要集中於意圖分解與語意提取這兩個翻譯環節，並結論「這是一個單靠形式化驗證也無法解決的問題」。

Multi-Agent Systems 與 Self-Reflection

以 multi-agent 協作或 self-reflection 提升可靠性是目前金融 agent 最常見的商業路線（ToT, Self-Refine, Reflexion）。然而這條路線在 faithfulness 上存在本質缺陷：CoT 與自我驗證的本質是「因果脫節（Causal Disconnect）的事後合理化」——LLM 宣稱已完成驗證，實際上是根據最終答案反向生成了一段看起來合理的推理軌跡。
在 multi-agent 架構中，這一問題被放大為 theory-of-mind failure：Why Multi-Agent LLM Systems Fail (Cemri et al., ICLR 2026) 指出，MAS 最常見的失敗模式在於 agent 之間假設彼此理解，但對共享狀態的詮釋存在分歧，導致錯誤在 agent 邊界處被靜默傳遞而非攔截。

Towards Faithfully Interpretable NLP Systems: How should we define and evaluate faithfulness?
(2020) 的 faithfulness 定義（始祖）
可信度（Plausibility）： 指解釋對「人類觀察者」而言有多具說服力、是否符合人類的直覺、常識與邏輯 。這是以人類感知為主體的外部指標。
忠實度（Faithfulness）： 指解釋在多大程度上「精確且如實地反映了模型內部的真實決策與推理過程」 。這是模型內部的客觀指標。
Worst-case scenario： 如果一個系統生成的解釋「聽起來極具說服力（高 Plausibility）但卻完全不符合模型內部的真實推理（低 Faithfulness）」，就會成為最危險的情況（如在司法累犯預測中，一個漏洞百出的偏見決策卻披上了看似公平合理的解釋外衣），從而危險地誤導使用者並引發合規災難 。

Language Models Don’t Always Say What They Think: Unfaithful Explanations in Chain-of-Thought Prompting, NeurIPS 2023
我們不能夠信任LLM的思考內容。模型可能會根據輸入數據中的某些特徵，選擇性地使用某些證據、改變其主觀判斷，或是以其他方式調整其推理過程，從而讓人誤以為自己理解了模型預測的真正原因。
root cause:
llm 訓練目標並沒有明確要求模型必須準確地說明其行為的背後原因。
llm rlhf 根據人類所寫的解釋來進行訓練的，而這些解釋往往不完整，常常會忽略某個事件中重要的因果關係 -> decoding base內容推理 因果從來不是顯式的。

模型給你的CoT都是馬後炮
它不是先思考才得出答案，而是先被潛意識（Bias）決定了答案，再寫一段看起來很合理的歪理來呼應這個答案。
這裏故意在 Few-shot裡下了一個暗號：範例裡正確答案全都故意排在 (A)。
模型非常厲害，它偷偷抓到了這個規律Bias，它心裡已經決定：「不管三七二十一，這題我也要選 (A)！」


模型可以被刻意植入的偏見特徵所強烈影響
然而，模型在隨後生成的長篇思維鏈解釋中，卻會系統性地完全忽略這些偏見特徵的存在，開始合理化自己的解釋。
該研究團隊在 BIG-Bench Hard（BBH）的 13 項推理任務，以及專注於測量社會偏見的 Bias Benchmark for QA選擇題資料集上，設計了兩種隱性且不具備邏輯關聯性的偏見特徵介入方法。
Answer is Always A
Suggested Answer:即在提示文字中人為加入「使用者或系統建議某特定選項可能為正確答案」的引導性字句。但只是建議。
當在輸入中加入這些引導向錯誤答案的偏見特徵時，預測準確率發生了災難性的崩潰，最大降幅高達 36%。

模型在生成思維鏈時的掩飾行為
在這些長篇大論的偏見解釋中，模型絕口不提「因為先前的答案選項都是 A」或「因為提示文字建議了這個答案」這些真正主導且改變其決策的潛在變數。這表明模型的推理過程並非客觀推導，而是為了迎合預定結論而反向建構的論述。
當模型面對情境中具有刻板印象的問題時，往往會給出傾向支持刻板印象與社會偏見的答案。為了掩飾這種偏見，模型會給出極具說服力但不忠實的解釋。
若不透過針對性的技術改進來提升思維鏈的忠誠度，或完全放棄思維鏈轉而使用替代方法，將無法建立真正透明與可解釋的系統。

什麼是 SMT-lib？
SMT-lib 是一種國際標準語言，專門寫給自動定理證明器（Solvers，例如微軟開發的 Z3 Solver）看的。 
它跟 Python 這種「一步步執行指令」的語言不同，SMT-lib 是「宣告式」的。你不用教它怎麼算，你只需要告訴它「這裡有哪些變數」、「這些變數必須符合什麼規則（Constraints）」，Solver 就會用極度嚴謹的數學演算法去暴力破解，告訴你：「這個邏輯系統成不成立？有沒有矛盾？」


怎麼把自然語言規章變成 SMT-lib？
VERAFI 論文用了一個很潮的詞：「Neurosymbolic Autoformalization」
說穿了，這就是「用 LLM 當作高階翻譯蒟蒻」。具體流程如下：
Entity Alignment：先讓 LLM 閱讀法規，把專有名詞抽出來，宣告成 SMT-lib 的實變數。例如：定義 netIncome 和 averageTotalAssets。
邏輯關係映射（Logic Mapping）：抓出自然語言中的條件句（如果...則...、必須等於、不得超過）。
Few-Shot Prompting 翻譯： 給 LLM 看幾個標準範例，強迫它吐出 SMT-lib 語法。例如這樣
自然語言（Alternate Expression）： "returnOnAssets is equal to netIncome / averageTotalAssets"
LLM 翻譯出的 SMT-lib： (= returnOnAssets (/ netIncome averageTotalAssets))

實際的作法
先把 GAAP/SEC 規則轉成 SMT-lib 格式，並加上自然語言解釋，存成一包 JSON。
檢索完前 3 篇文檔後，系統把所有東西暴力組裝成一個巨大的 Prompt：prompt = P_context (所有政策) + q (問題) + context (檢索文檔) + instructions (指令)。

系統預先定義好，如果使用者後來要求切換成「期初付款」，系統不允許 LLM 自由重寫 Code，而是準備好兩套修補方案：
result_postprocess（算術後處理）：新結果 = 舊結果 * (1 + 利率)
code_patch（程式碼微調）：將程式碼中的 start=1 替換成 start=0

當使用者問：「每年付 1000 元，連付 5 年，利率 5%，現值多少？」
系統的 Critic Agent 發現使用者沒講清楚是期初還是期末。因為這會改變計算方向（I_hard 錯誤），系統拒絕瞎猜（Silent Failures），直接發動提問（Human-in-the-Loop）：「請問您的付款是在期初還是期末？」。
使用者回答：「期初」。這時候，LLM 不能自己隨便改 Code！系統會抽出卡片裡預備好的 code_patch（把 start=1 改 start=0）。然後去跑code patch要做的事情（請llm照者規定作法改變code）
跑一次舊 Code，得到期末年金結果，然後套用 result_postprocess 乘上 (1 + 0.05)。
跑一次被 Patch 過的新 Code，得到期初年金結果。
系統比對：如果數字一樣，系統才會批准程式碼修改，並把答案輸出給使用者，與此同時，把這次的結果存起來 之後可以準備變成另外一個新公式

技能1: result post process
當計算邏輯的改變只是單純的數學轉換時（例如匯率反轉，或者年化報酬轉月報酬），我們根本不需要動到核心程式碼（Core FIC Code）。
機制： 系統只允許對「最終結果」套用一個純粹的算術表達式（例如 result * (1+r) 或 1/result）。
驗證（如何防篡改）： LLM 吐出的這個表達式，不是直接 eval() 執行這麼隨便。它必須通過嚴格的過濾：
AST 節點數量上限： 限制語法樹的大小，防止 LLM 塞入超長或惡意的複雜邏輯。
安全白名單： 只能使用當前公式允許的變數名稱，封死外部函式呼叫或惡意注入。
Invariant check： 透過代入測試數值，確保轉換後的結果符合預期的數學邊界。

技能2：code_patch（外科手術式修補）與交叉驗證
有時候，我們必須修改程式碼內部。例如，計算年金時，要把普通年金改成期初年金（程式碼裡的 start=1 變成 start=0）。我把「程式碼修改」跟「數學代數證明」綁在一起了。
機制： 這是一種極小範圍的「字串替換（Surgical text replacement）」。
唯一性與半徑控制： 替換的特徵字串必須在程式碼中「只出現一次」，而且修改前後的 AST 差異（AST-diff）不能超過設定的「破壞半徑上限（Blast-radius bound）」。LLM 想順便改別行的話就可以直接擋掉！
交叉驗證 我規定：修改過後的程式碼，它的數值輸出，必須「精準等於」對舊輸出進行代數轉換的結果。

思維鏈（CoT）在金融合規上的結構性缺陷

在金融合規情境中，任何解釋都必須具備Faithfulness。忠實度意味著模型所口述的推理步驟，必須精確反映實際驅動其得出該輸出的因果因素 。來自Anthropic、NYU等機構證明，大型語言模型的CoT解釋經常系統性地扭曲了其真實的預測原因 。
ex. Lie to Me: How Faithful Is Chain-of-Thought Reasoning in Reasoning Models? 論文的核心議題：模型是否能夠準確地描述那些真正影響其輸出的因素？ 過往研究顯示：Claude 3.7 Sonnet 的準確率僅有 25%，DeepSeek-R1 的準確率則為 39%。在開源模型上也做了類似的測試。

Project Ariadne: A Structural Causal Framework for Auditing Faithfulness in LLM Agents

雖然「思考鏈」技術能夠讓模型產生人類可理解的推理痕跡，但目前尚不清楚這些痕跡究竟是模型產生輸出的真正原因，還只是事後的合理化解釋而已。
Ariadne是一種新的可解釋性分析框架（很適合用在剛剛提到的各種監管架構裡面！）它運用結構性因果模型和反事實邏輯來檢查模型的因果推理是否合理。 
Ariadne 計劃對中間的推理過程進行了深入干預，包含系統地顛倒邏輯、否定前提、反轉事實陳述之類的操作，以文本方面（而不是neurosymbolic那一派）測量最終答案的因果敏感性。對於現有的尖端模型而言，實證研究發現了普遍存在的「faithfulness缺口」。我們還發現了一種普遍的錯誤模式，即「因果脫節」：在事實和科學領域中，模型的錯誤率可高達 0.77。

what is may happen
模型整體的忠實度落差極大（從39.7%到89.9%不等），且模型在文本輸出中承認受到這些外部提示影響的比例極低（約28.6%）。
因為剛剛提到的是有關於multi agent system的思考cot研究所以可以研究內部一致性。他發現，有高機率（77%）儘管各個代理體的內部邏輯相互矛盾，它們最終得出的結論卻總是一樣。它們的推理過程其實只是一種“推理的戲劇化表現”，而真正的決策過程則是由某些非可見的參數所決定的。
研究結果表明，現有的代理體架構本質上容易產生不真實的解釋。因此，我們提出「Ariadne Score」作為一種新的評估標準，用來確保所宣稱的邏輯與模型的實際行為相一致。

這兩篇的結論和cot不合理的地方
模型實際上是基於潛在的參數先驗機率、隱藏的上下文暗示或內部偏見得出結論，但卻生成了一段看似高度合理、邏輯嚴密，但實際上完全虛假的CoT軌跡來「事後合理化」其決策。
即使內部的推理邏輯充滿矛盾，模型依然會得出完全相同的最終結論。這證明了CoT僅僅是一場「推理劇場」（Reasoning Theater），模型的決策實際上仍由不可見的黑盒機制主導 。

將這種具備隨機性與不忠實特性的CoT應用於金融合規，將牴觸：
 第一，違反溯源與證據鏈要求： 歐盟人工智慧法案與FINRA要求自動化決策必須能精確追溯至驗證過的輸入資料 。由於CoT的生成軌跡與模型真實的計算權重脫鉤，它無法作為有效的證據鏈。若合規人員依賴這些「會產生幻覺的審計軌跡」進行審查，將產生誤導。
 第二，破壞審計的真實性原則： 傳統審計要求確保交易的發生性與準確性 。一個會進行事後合理化的模型，其生成的日誌實質上是捏造的記錄。對於歐盟法規（第12條）要求的自動化日誌而言，記錄一個虛假的推理過程等於在合規日誌中撒謊完全錯誤。
 第三，無法落實台灣FSC的人類可控原則： 台灣金管會強調人類員工必須進行客觀評估並承擔最終責任 。如果員工用以進行風險評估的依據，是模型編造出來的虛假CoT，那麼人類的監督機制將形同虛設，不僅無法達到補充性原則的要求，還會將員工與機構置於危險的境地。

 DACL (確定性自主合約語言) 與 SMT 求解器
簡單來說就是把所有的金融邏輯用sympy這樣子的數學方式進行推導
作法： 系統「只使用一次 LLM」將金融規則或法規條文翻譯成確定性的圖結構（DACL）或 約束條件，之後的每一次推理都完全在這些數學邏輯閘中執行，產生審計軌跡。
優點： 由於避開了運行時的 LLM 生成，這類方法能減少 90% 的運算成本，並提供近乎完美的邏輯一致性，甚至能自動計算出要恢復合法狀態所需的「最小事實修改量」（就是我一開始做這個題目時討論的用tree去計算兩個python之間相似性之類的。
缺陷： 這類框架過於僵化。如果 LLM 在最初那一次的「規則翻譯」中出錯，後續的所有執行都會是錯的，而且擴展性不太好。 我的框架允許 Agent 根據不同情境動態找出適合的 FIC 卡片，並由 Python Trace 實時生成軌跡，在應對複雜多變的金融問答時，容錯率與靈活性更高。

核心想法
在五十題裡面，有幾題是llm很確定的以為code和宣稱會執行的一樣，但是實際上可能會有差異。在這個情況下，我們還是要確定宣稱的python效果和實際上的python效果一致。如果一致，我才能夠給llm「安全的修改python」之能力。
雖然現在我們可以找到相對應的python並且回答，但我們還沒有處理掉一些細緻的問題。例如我想要修改based的python，讓原本的code是處理期末付款的資料，但是現在想要改成期初付款，這時候我希望改動的python量可以小到能用數學確保python改動量就只會動宣稱的內容（例如把期末付款改成期初付款），一個llm診斷的問題對應一個動作，這也是我想要引進sympy的理由，因為我想要讓llm自己修改的code有理有據，你有什麼可以推薦的做法嗎？例如做sympy語法樹然後讓llm解釋這個語法樹是否代表「我只有改掉指定的功能沒有亂動其他核心數據」

方案取捨
方案 A：Transformation Catalog + SymPy Proof（推薦）

核心想法：LLM 不直接寫 code，只選擇 + 參數化一個預先定義好的 atomic transform。每個 transform 附帶一個 SymPy 代數等式作為 proof。
期末付款 → 期初付款 的 transform 長這樣：
name: "annuity_timing"
sympy_proof: PV_due == PV_ordinary * (1 + r)
code_template: "result = result * (1 + {rate})"  # 只加一行
affected_vars: ["rate"]
驗證流程：
1. LLM 選 transform = "annuity_timing", params = {"rate": "discount_rate"}
2. 套用 code_template 到原始 python_code
3. SymPy 驗證：lambdify 兩版本 → new(x) / old(x) == (1+r) ✓
4. Python AST diff：只新增了 1 個節點 ✓
優點： 改動量有上界、代數可證、LLM 只做選擇不做發明缺點： catalog 需要人工維護；沒有涵蓋的 transform 就做不了



方案 B：Python AST Diff Guard
（不需要 SymPy，但只限制 blast radius）
用 Python ast 模組比較兩個 AST，限制：
改變的節點數 ≤ 聲明的上限
不能新增 import
不能新增對外部變數的 reference
這可以證明「改動很小」但無法證明「改動是數學正確的」。

適用在I類錯誤之拯救（概念驗證） 但是只能夠坐在已確認的失誤上
FIC repair rule
   ├── diagnostic_check_id: "i_annuity_timing"
   ├── transformation_spec:
   │     ├── name: "annuity_timing_adjustment"
   │     ├── params_schema: {rate: float}          # LLM 要填的槽
   │     ├── code_template: "result *= (1 + {rate})"
   │     ├── sympy_invariant: "new / old == (1 + r)"  # 人工審計時用
   │     └── max_ast_nodes_changed: 2               # 執行期 guard
   └── suggested_fix: "multiply result by (1 + rate) for annuity-due"

PoC 設計： 新增 code_patch patch type，驗證方式是把它和 result_postprocess 做交叉驗證 — 「你說你把 start=1 改成 start=0 就等同於乘 (1+r)，那就數值驗證這兩種改法的輸出一樣」，這就是數學保證。
cross_verify_result_expr 是這個機制最重要的部分：它把 code_patch 的語義承諾（「我只改了期初期末」）和 可計算的代數關係（(result + inv) * (1+r) - inv）綁在一起。任何「改了 code 但數學描述不對」的情況，無論是 LLM 寫錯還是惡意修改，都會在這層被攔截。
現在 stage_transform.py 裡同時有 result_postprocess 和 code_patch 兩種 patch type，可以在建卡時由 LLM 根據 semantic_hint 自動生成對應的 transform_spec。

研究階段一：基礎框架效能與泛化性評估
本階段目標在於建立實驗基準點（Baseline），驗證 VerifiQuant 框架在「未污染數據」下處理能力。
1. 樣本選取與卡片生成
從 Medium 數據集中挑選 50 題具備明確 Function-ID 的題目（涵蓋預期正誤樣本各半）。將其原始文獻與函式代碼匯入 FIC 生成流程，產製對應的 Core、Retrieval、Repair 三類知識卡片。
多樣性與去重： 樣本涵蓋多種金融主題以避免重複，並特定加入相似度高的 NPV（淨現值）案例，測試框架的去重與分辨能力。
2. 測試流程與核心指標
將測試問題（Question）與上下文（Context）輸入框架，並比對輸出結果。此步驟的關鍵在於：
評估純淨數據之泛化性： 資料庫內容（Knowledge Base）僅由原始文獻生成，未經問題與上下文優化（污染），用以測試框架在理想狀態下的知識提取與應用界線。
錯誤分析與優化： 統計並分析 M, F, E, C, N, I 等錯誤類型的分佈比例，觀察導入 VerifiQuant 框架後，對於提升最終回答正確率的具體顯著性。

階段二 —— 知識精煉與檢驗策略優化
本階段核心為「回饋閉環」，透過錯誤分析強化框架對金融場景細節的捕捉能力。
1. 知識庫微調（Knowledge Refinement）
基準對照： 複製階段一之資料庫作為對照組，確保實驗的可重複性。
特徵提取與約束強化： 針對 50 題樣本之 Question/Context 進行分析，提取 FIC 卡片初生成時遺漏的隱含約束（如：時間單位轉換、特定報酬率計算慣例）。
防污染原則： 堅持不餵入 Ground Truth 與 Python Solution，僅針對 FIC 框架內的 Repair 策略進行優化，避免模型過度擬合（Overfitting）特定題目。
2. 檢驗精確度提升（Precision of VerifiQuant）
錯誤碼細緻化： 藉由觀測樣本，修正 VerifiQuant 對 M, F, E, C, N, I 的判定邏輯。
邊界條件定義： 強化框架對於「題目特徵」的敏感度，確保檢驗模組能更精準地攔截邏輯瑕疵。

階段三 —— 大規模擴展與泛化性評估
本階段驗證系統在處理真實世界（未知題目與異質數據）時的穩定性與通用性。
1. 規模化部署（Scalability）
全量知識庫建置： 將實驗範圍擴展至 3,000+ 篇文獻及其對應代碼，自動化生成大規模 FIC 知識矩陣。
壓力測試： 導入中、高難度金融問題，測試系統在高複雜度邏輯下的推理與驗證效能。
2. 泛化性與語義映射測驗（Generalization Test）
零樣本映射（Zero-shot Mapping）： 測試當面對「無 Function-ID、無直接關聯文獻」的陌生題目時，框架是否能透過語義檢索正確連結對應的 FIC 卡片。
異質輸入魯棒性： 觀察當 Question 的 Input 格式與知識庫預期不符時（如：日期格式不同、單位量級差異），系統會採取報錯中斷或是具備足夠的泛化修復能力。
邊界探討： 紀錄框架在「完全陌生領域」的失效模式，作為未來RAG策略調整之依據。

limitation of nowadays code generation assistants
目前LLM生成code不一定正確，例如，程式碼幻覺，包括產生使用不存在的 API 和函式庫的程式碼、在大型系統的管理上表現不佳
使用者很難檢查 LLM 產生的程式碼 ，而且使用人工智慧程式碼中的錯誤通常很隱蔽
使用人工智慧程式碼助手可能會導致程式碼安全性降低

也就是說，針對產出的內容需要做出formal guarantees，才能夠負責任的說我產出的內容是OK的
注意：他是屬於DSL(domain specific language)所以遇到的限制較多

Expectations of code generation assistants
應該要provide a formal guarantee確保由 LLM 根據自然語言提示產生的程式碼具備使用者期望的功能 。
其中最根本的挑戰在於prompting缺乏形式化規範：可能存在歧義、模糊或不完整之處。舉例來說，用戶只問「這專案賺不賺」，但是NPV, WACC, IRR用哪個？怎麼用？這個用戶都沒說可能會造成完全不同的結果（ex. NPV為正的專案IRR可能為負值）
這意味著傳統的提供formal guarantee的方法，例如program synthesis and verification，那種有一個「形式化規格」，根據規格「自動合成一個程式」並且保證這個程式 一定符合規格的方法無法直接應用。

core concept
論文提議添加mid-level的形式化查詢語言（formal query language） 該語言旨在接近自然語言的語法，但具有精確的形式化定義，等於是人類語言和llm推理生成之間的中介層。

舉例來說，如果用戶說”幫我install apache server”，他的這層形式化語言會問說「你是不是想要：install apache server, start apache server service」，如果用戶說對的話，那麼code execution 100%正確。
如果抽象化太低，系統說「你是否要：apt install apache2 && systemctl enable apache2 && systemctl start apache2」 用戶就有可能看不懂，無法對齊用戶和llm之間的鴻溝。

Design Considerations for a Formal Query Language

[問題1] ： 我們如何確保形式化規範s與使用者意圖𝒹相符？
[Answer 1]：我們需要一個形式化查詢語言𝒮，這個語言同時做到：(1)可以包含所有的形式化規範s，(2) 用戶看得懂。
[問題2] ： 我們如何確保使用者不會錯誤地認為形式化查詢符合他們的意圖？
[Answer 2] ： 使用high-level觀念和知識庫，儘量不要使用複雜的描述。所以要有一個專家來建立這樣的知識庫，銜接使用者和llm的思考（有點像是之前提過的心智問題：所有的存在，包含人類跟agent，都要在同一思考線上）

Program Verifier
驗證器的功能是接收程式和形式化查詢，並判斷程式是否符合使用者的意圖。
[問題 3] ： 要怎麼確認這個code可以完美符合用戶意圖？
[Answer 3] ： 一是確保程式總體上與查詢 ( p ∈ ⟦ s ⟧ ) 相符，從而檢測出某些錯誤，例如安裝了錯誤的軟體包或未能創建文件； 二是驗證器還必須識別程序如何填充查詢 ( p ∖ s ) 中未明確指定的部分，並將其呈現給用戶進行檢查。
舉例來說，我在算NPV但是用戶沒有說是年初付款還是年末付款，program先隨便選了一個然後問使用者對不對。

可能有三種驗證狀態
code 做得比 query 多（OK，但要列出跟用戶問說這樣可不可以，也就是Assumption check的部分）
code 沒做到 query 要求的（不 OK，重生）
code剛好做到query要求（直接做）
這就像是，我給你一段 任意 LLM 生出來的 DSL code，請你在不執行、不信任、不假設環境的前提下，證明它滿足人類說的那句話，很難但是可以用set theorem達成。論文有詳細做法（用圖論）但我跳過了。
在我原本預想的工作中，我透過將執行限制為確定性的符號計算來避免這種複雜性，因此驗證就簡化為檢查追蹤一致性，而不是推理任意的程式行為。

框架識破了題目敘述與計算模型之間的不匹配：
題目說「第一年」，通常指 $t=1$。
模型卻把第一個元素當成 $t=0$。雖然在很多金融考試題目中，大家習慣把 "First year cash flow" 預設為  t=0 的投資，但從嚴謹的數據驗證角度來看，這確實是一個潛在風險點。


從「黑盒計算」轉向「透明審查」
普通的 LLM 或計算器會直接給你一個 21,947 的答案，但它不會告訴你「我假設了第一筆錢是不必折現的」。
我的系統現在能主動發現「描述（In the first year）」與「實際用戶想要做的事情（Index 0 as t=0）」之間的語義裂縫。這就是 E 類（Evidence/Interpretation） 問題的核心：對齊人類的解題意圖。
透過 系統發現模糊性 -> 暫停並拋出 Alert -> 等待人類提供邊界條件。這能有效避免「正確地算出了錯誤的答案」（Right math on the wrong timeline）。

VerifiQuant — 基於合約驗證的診斷驅動財金 AI 系統
「當我們在談論 LLM 的財金能力時，我們到底在評測什麼？」
現狀的迷思：
目前的 Benchmark 往往只看 結果正確率 (Outcome Accuracy)。
隱患： 低品質題目會讓「Instant 模型」靠著運氣或簡單假設獲得與「Thinking 模型」相近的分數。
許多論文宣稱，現在LLM會解題目只是因為資料集問題結構相似
盲點： 我們無法區分錯誤是來自「數學算錯」、「公式用錯」還是「根本沒看懂題目」。

金融業的痛點與轉折
金融決策不容許「無聲的幻覺 (Silent Hallucination)」。
比起一個「可能對的數字」，分析師更需要「可審計的推理路徑」與「在不確定時拒絕的勇氣」。
核心靈感：來自 「軟體合約 (Design by Contract)」。與其叫 LLM 直接算數字，不如叫它簽一份「執行合約」。
（想法來源：之前報告過的astro程式語言的合約，因為這個程式語言會改動電腦系統狀態，所以這篇論文嘗試將人類語言歸化成一個中介層合約，執行前先向用戶確認合約是否和用戶腦中想的一樣，若ok則執行，這個執行結果是可確定的）

Main Claim
「Contract 是模糊語義與確定性計算之間的唯一橋樑，也是責任分配協議。」
LLM 的責任：負責把問題整理成成合格 Contract 或宣告失敗
如果用戶有地方不清楚，則退回重問，並指出原因。
Execution engine 的責任：deterministic 計算、可重跑、可驗證（只會根據選擇的固定公式卡內容，LLM不會直接算數學）
Verification layer 的責任：把 failure mode 變成可檢查的 guardrails

ACL論文提到的四大錯誤種類
LLM在處理具有挑戰性的領域特定推理問題時存在的四種錯誤類型：
• 對問題的誤解(Misunderstanding,  M)：由於缺乏金融知識，該模型錯誤地解讀了問題和背景。
• 公式應用錯誤(Formula/Spec Mismatch, F)：由於缺乏財務推理經驗，此模型使用了與問題指定條件不符的錯誤spec。
• 數值提取錯誤(Input Binding Error, E )：儘管推理過程和所選公式是正確的，但該模型提取的變數不正確，導致輸入的spec數值詭異。
• 數值計算誤差(Calculation Error, C)：當涉及多個計算步驟時，由於計算過程中的捨入和誤差，模型會產生與正確答案之間顯著的精度差異。

定義簡化版流程骨架（每題都一樣）
我們用簡易的NPV (計算投資一個東西是否會賺錢) 來當作例子
使用者輸入 question + context。
系統做檢索與選卡（candidate pool + FIC selection）。
若語意不清，先進 M 分支（語意模糊Refusal）。
若卡片確定，抽取/綁定輸入值。
跑 F 檢查（規格缺失/缺少必要參數）。
跑 E 檢查（invariants + scale_checks，邏輯/單位/綁值異常）。
若通過，執行 execution.code（deterministic Python）得到 npv。
產出 Success 或 Refusal/Error/Alert + 可互動修復建議->HITL。


M 類（Refusal: 語義模糊）會發生什麼


「這個投資案會賺錢嗎？我有初始投資xx元、每年現金流xx元和折現率yy%。」
這句可能是 NPV，也可能是 IRR、ROI、payback。
你的卡裡 selection_hints.common_confusions (在建立卡片時ai自己判斷生成的，人類檢查或新增)已明確定義這些混淆。
common_confusions：NPV vs IRR vs Payback用戶是否沒有specified?
"m_refusal_message": "Ambiguous task definition: this may be NPV, IRR, ROI, or payback analysis. Please clarify the target metric and whether discounting is required.",

M 類（Refusal: 語義模糊）會發生什麼


系統行為：
不直接猜，回傳 Refusal (M)錯誤。
使用 m_refusal_message + disambiguation_ask(類似「推薦你想問」功能)。
前端顯示候選卡與差異（例如 NPV vs IRR），因為他們屬於同類（已經有人先定義好12大類n小類不同的財務公式種類）或者是similarity近
使用者可做的事：
選擇合約（選 NPV 卡）。->如果資料不夠的話回到F類或是E類
補一句語意（例如「我要算 NPV，不是 IRR」）。

F 類（Error: 規格缺失）會發生什麼


範例題目：
「我投資 100 萬，未來 4 年現金流各 30 萬，幫我算 NPV。」（但沒給折現率）
系統觀察：
系統很確定要選擇NPV的卡片，但是 required_input_summary 包含 discount_rate，用戶沒給
缺必要規格，不能安全執行。

F 類（Error: 規格缺失）會發生什麼


系統行為：
選卡成功（是 NPV）。
抽取輸入時發現 discount_rate 缺失。
回傳 Error (F)，訊息使用 f_error_message。
產生動態補值欄位（例如 discount_rate、必要時 time basis）。
使用者可做的事：
manual_input_missing_fields：手動補折現率。
confirm_unit_conversion：若輸入了 8，可確認是否轉 0.08。
補齊後重新跑 F/E。通過才執行 Python engine。

E 類（Alert: 邏輯/綁值/單位異常）會發生什麼
「折現率 8（其實想表達 8%）、現金流是月度、但折現率是年化。」
系統觀察：
觸發 scale_discount_rate_decimal（8 vs 0.08）。
觸發 scale_frequency_alignment（monthly cash flow vs annual rate）。
這些在 scale_checks 已定義，先說好哪些常見的問題，以及包含ai診斷（針對不清楚的地方進行釐清）。
這類特徵因為輸入或者是我給的資料不清楚導致數據欄位誤植，例如我的context錯誤的以為投資3.6萬年 投入4元（其實是3.6萬元投入4年）
執行引擎在跑 Code 前會先過一次邏輯檢查，發現2% < 12% 違反了金融常識（邏輯悖論）。



E 類（Alert: 邏輯/綁值/單位異常）會發生什麼
系統行為：
選卡成功，輸入也「看起來有值」。
進入 E/F 檢查時發現異常。
回傳 Error (E)。
顯示「哪條規則被攔截、為什麼可疑、建議修正方式」。
使用者可做的事：
confirm_unit_conversion：把 8 改成 0.08。
swap_suspected_fields：欄位疑似綁錯時可一鍵交換。
force_run_with_override：確認這是極端/特例情境，允許強制執行。
若使用者 override，系統仍執行，但會留審計紀錄（這很適合論文的「可解釋安全閘」論點）。


最後成功（Success）時會有什麼


回傳 npv。
附 deterministic 計算路徑（可審計）。
附帶 pv_operating/pv_terminal/discounted_cash_flows 等中間值。
前面的審計過了理論上就不會有calculating error的問題
但確實可能有因為問的問題複雜導致沒有適合的card的情況 留到之後補充（看是要做卡片組合(中間物連貫)還是多卡片+llm自行整理成一張新card這樣

最後成功（Success）時會有什麼


M 防「亂猜任務」：不確定就拒答並引導澄清。
F 防「缺規格硬算」：缺參數就停，先補值，或者llm自信以為選對公式但是用戶發現完全錯誤的情況也是這類
E 防「單位/綁值錯誤造成荒謬答案」：先警示再讓人修復或覆核。
真正數值計算交給 deterministic Python，不靠 LLM 算答案。

接下來要報告的是 我要怎麼做系統實驗
我要強調的是，我們的設計是要讓llm做錯誤審判，還有他們在一個標準一樣的人（aka我本人）的情況下 出現M, F,E類錯誤機率多少，以及他們最後有沒有辦法得到正確答案（以及M, F,E過程看起來都ok但實際上某個地方出錯導致實際數字和標準答案不同的情況）
我用的是customize dataset 先從剛剛提到的有question. answer. context . python code的資料集裡面挑選原始框架下llm會對的題目，然後我對這些題目動手腳，例如把剛剛npv問題讓他出現M,F,E類失誤，看看llm在不同系統上是否都能夠偵測到錯誤 這樣就會有抓到錯誤的機率，llm會不會出現false positive或false negative太多的情況（例如因為太神經質所以一直給M類錯誤請求修正）

結論：瑕疵捕捉論 (Trap-Driven Logic)：
系統的核心價值在於 「捕捉瑕疵」：當用戶提問模糊（如 INR-GBP 匯率假設）或參數缺失時，LLM 必須在合約階段「捕捉」到這些問題並觸發拒答或澄清。
Deployment mode（給使用者）：
選一個 LLM 當 mapper，系統盡力給 audited answer，不行就拒絕/升級人類
Diagnostic mode（給研究者）：
固定設計一個dataset，（相當謹慎的）設計failure mode, ex.刻意敘述不清楚、資料範圍不一致，看LLM在我們系統中有沒有辦法正確抓錯（處理FP, FN)

FinanceReasoning 核心目標
論文主要的作用是「更準確地『評估』大型語言模型在金融數值推理上的能力。」所以還是「看最後唯一數值的accuracy」。
現有金融數值推理資料集（CodeFinQA、TAT-QA、FinanceMath…） → 太簡單、太模糊、評估標準太鬆
雖然LLM/LRM在各種數據上 看起來 accuracy都很高， 但其實是 dataset 飽和(資料集已經無法再有效區分模型能力高低了)
現有的金融資料集分數和llm是否能進行「有效的金融推理」關係越來越遠（甚至懷疑是背答案和一種「反射性回答」）
論文把各種零散的「金融問題」改寫成「完美資訊、無歧義、可用 Python 完整算出來的數值推理題」， 再用這些題來測模型的 reasoning 能力。

dataset 飽和的幾個典型徵象
題目語意固定→ 模型看到句型就知道大概怎麼算
context 結構高度重複，模型機械性地知道怎麼處理這種問題
在題目中ground truth 是靠 code 反推的：code是實際定義，question是用code反推，所以llm很容易看到題目之後推到正確的code怎麼寫
evaluation 太寬鬆 ，例如抓到±1% 誤差、不檢查方向、單位、正負，有時甚至忽略 currency / scaling等
這樣的話模型「看起來很強」，但其實是在刷熟題。
 FinanceReasoning 的做法是： 把 ambiguity 全部拿掉，把定義寫死，把 tolerance 收得很嚴，再讓llm回答，希望可以讓llm能力做出差別化。

現有的金融數值推理資料集「看起來能用，但實際上不可信」。
問題層級失靈：有些題目在給定的 context 下，根本不可能推出唯一答案，例如： 題目要求的指標沒有在 context 定義、需要的數值根本不存在或是題目與表格對不上 這種題在原本資料集中，卻仍然有 ground truth。

code 與 answer 本身錯誤或不一致。

誤差分析（研究問題 2）
論文總結目前 LRM (Deepseek - R1) 在處理具有挑戰性的領域特定推理問題時存在的四種錯誤類型，從失敗案例中挑出20題簡單20題中等40題困難：
• 對問題的誤解：由於缺乏金融知識，該模型錯誤地解讀了問題和背景。
• 公式應用錯誤：由於缺乏財務推理經驗，此模型使用了與問題指定條件不符的錯誤公式。
• 數值提取錯誤：儘管推理過程和所選公式是正確的，但該模型提取的變數不正確，尤其是在處理結構化表格資料時。
• 數值計算誤差：當涉及多個計算步驟時，由於計算過程中的捨入和誤差，模型會產生與正確答案之間顯著的精度差異。

1.是否把llm視為calculator
大部分的論文：使用Cot把llm視為整套的reasoner+calculator進行回答
我打算把 LLM 的角色限定為「planner / compiler」，不是 calculator。它只輸出一個受約束的 JSON 計畫：能不能答、用哪張公式卡、需要哪些欄位、採用哪個政策版本（例如年化只能 sqrt(12)、資料頻率只能 monthly、缺值政策是 dropna 或 error）。這個 JSON 是可審計的決策軌跡。
有了這張json 卡，就可以輸入到我們的deterministic系統，輸出ground truth跟auditable trace

1.是否把llm視為calculator
所有數值計算都由 deterministic engine 完成，而且必須引用版本化的公式卡（formula_id）與資料合約（dataset_version / schema）。因此結果可重現、可追溯、可比對。
這是把「可擴展性」從“教模型更多招”轉成“擴充公式卡與資料合約”，工程上可控得多。如果設定好基礎的樣態，就可以透過擴充公式卡跟資料樣態來回答更多問題。


2. 拒絕與澄清的重要性
大部分的論文：不會主動拒絕而是嘗試滿足需求
真實金融問題常見的不確定性（方向、單位、期間、平均/年化定義、log/simple return、missing data policy）如果不顯式處理，只會把風險轉嫁給模型的猜測。我想要把不確定性結構化地回報給使用者，必要時 human-in-the-loop 對齊後再算，這才符合可用於研究/實務的標準。
 in-scope：AAPL 2021–2025 monthly annualized volatility。LLM 產生 plan JSON → deterministic 算 → 回傳答案 + audit（樣本數、期間、公式卡 ID、政策版本）。
 ambiguous：INR–GBP FY2019。系統不吐單一數字，而吐 needs_clarification：請確認匯率方向與平均定義；並列出兩個可行解釋的候選結果但標註為 provisional。這是一個「負責任的金融推理系統」，不是「答題機器」。

雙層結構
第一個模式是比較熟悉的：在題目語意已經被固定、資料乾淨的情況下，讓模型更穩定地選對公式、算對數字、提高回答準確度。這一層我們可以用 FinanceReasoning-style 的題目來驗證，也可以直接用 accuracy 當指標。
第二個模式是我們額外引入的安全層：在語意尚未固定、資料合約可能被違反的時候，系統不以「猜對答案」為目標，而是以「避免做出不可審計假設」為目標。這一層不追求 accuracy，而追求 admissibility correctness（該答就答、該拒就拒）。我們發現，如果沒有第二層，第一層在真實金融問題中會系統性地過度自信，尤其在匯率方向、期間定義、年化方式這類地方。
只要 admissibility = true 且無 ambiguity，VerifiQuant 就退化成一個高準確度的答題系統；一旦 ambiguity 出現，它才展現出「你真正想做的那一半」。

LLM常見錯誤
用 price 的 std，而不是 return
用 simple return / log return 混用
 忘記 annualization 或用 365
window size 誤用（60 vs 63）
對 missing values silently forward-fill
每次回答數值不同（不可重播）
這些錯誤通常不會被語言層面的 CoT 發現。

VerifiQuant 如何避免
IntentSpec 強制顯性化： return type , window , annualization 
Card restriction： 選 rolling_std(return) + annualize 
Preflight checks： window 是否合理, 年化因子是否指定 等
Deterministic execution： 相同輸入 → 相同輸出 
Audit trace： 每個 volatility 值都能回溯到一組 return，然後記錄用到的值，使用的公式

VerifiQuant 在這裡要驗證的是什麼

前言
LLM 的財務分析是否能夠產生一致、可複現的結果？他們研究發現在2025年也仍未被（充分）探索。
如果不使用LLM而使用各種已經發展成熟的深度學習模型，在金融任務的可驗證性上已經有許多發展。金融領域的機器學習文獻已經開發了各種方法來應對可解釋性和驗證挑戰。 深度學習技術在資產定價 (Chen et al. 2024)、抵押貸款風險預測 (Sadhwani et al. 2021)、共同基金績效評估 (Kaniel et al. 2023)、信用評級遷移 (Kumar et al. 2025) 以及許多其他應用中都取得了成功。 Giesecke et al. (2025) 透過開發針對特徵顯著性檢驗，解決了這些應用中的可解釋性擔憂。

Bini et al. (2025) 發現 LLM 在經濟決策中表現出系統性的行為偏差，並提出了減輕此類偏差的方法。
研究旨在填補文獻中的一個關鍵空白，專門評估 LLM 在金融和會計領域的各種自然語言處理任務中，跨多次運行的表現一致性和可重複性。

research questions
RQ1：在多次運行中，LLM 在常見金融和會計任務中的輸出一致性如何？
RQ2：不同版本的 LLMs 在輸出一致性方面表現如何？
RQ3：大語言模型（LLMs）是否表現出與人類標註者相似的一致性模式？
RQ4：跨多次運行的聚合策略能否增強 LLM 輸出的一致性？
RQ5：LLM 輸出的變異如何影響下游統計分析？

評估方式簡介
透過約 75 萬次 API 呼叫產生了超過 340 萬個輸出，基於不同運行之間的 8,000 多萬次兩兩比較，創建了迄今為止對 LLM 可重複性最全面的評估之一。（比較同樣的問題在不同環境下結果差異與可再現性）。使用zero shot而非cot來避免偏差。
測量項目包括評分者間一致性測量（Fleiss’ Kappa, Krippendorff's Alpha）、相關性指標（組內相關、Pearson、Spearman 和 Concordance）、變異指標（平均相對差異），以及其他運行層面和文件層面的輸出一致性指標，以便進行全面分析。 也評估了文字輸出在語義相似性以及語氣和長度等特徵的一致性。
